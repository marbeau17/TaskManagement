# サブタスク管理 仕様書 — PM視点のワークスプリットと工数配分

**作成日**: 2026-04-03
**視点**: プロジェクトマネージャー

---

## 1. 現状の課題

### 1.1 PMが直面する問題
- サブタスクに個別の見積工数・締切が設定できない
- 親タスクの工数がサブタスクの合計と連動していない
- サブタスク間の依存関係が管理できない
- 誰がどのサブタスクをどのくらいの時間で完了するか俯瞰できない
- 親タスクの進捗がサブタスクの完了率から自動計算されない

### 1.2 あるべき姿
PMがタスクを受けたとき:
1. タスクを複数のサブタスクに分割
2. 各サブタスクに担当者・見積工数・締切を設定
3. サブタスク完了に応じて親タスクの進捗が自動更新
4. 工数の合計が親タスクの見積を超えていないか確認
5. チーム全体のワークロードを考慮してアサイン

## 2. 機能仕様

### 2.1 サブタスク拡張フィールド

現在のサブタスク作成: タイトル + 担当者のみ
あるべき姿:

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| タイトル | text | ✓ | サブタスクの作業内容 |
| 担当者 | select | ✓ | アサイン先メンバー |
| 見積工数 | number (h) | ✓ | このサブタスクの見積時間 |
| 締切 | date | - | 個別締切（未設定時は親の締切を継承） |
| 優先度 | P1-P5 | - | 個別優先度（未設定時は親を継承） |
| 説明 | textarea | - | 作業の詳細・注意事項 |
| 依存先 | select | - | 前提となるサブタスク |

### 2.2 親タスク自動計算

**進捗の自動計算:**
```
親タスク進捗 = Σ(サブタスク進捗 × サブタスク見積工数) / Σ(サブタスク見積工数)
```
加重平均で、工数が大きいサブタスクの進捗がより大きく反映される。

**見積工数の整合チェック:**
```
親タスク見積工数 vs Σ(サブタスク見積工数)
- 一致: ✅ OK
- サブタスク合計が大きい: ⚠ 親の見積を上方修正するか確認
- サブタスク合計が小さい: ℹ 未分割の作業がある可能性
```

**ステータス自動更新:**
- 全サブタスクが done → 親タスクも done に変更提案
- 1つ以上が in_progress → 親タスクを in_progress に
- 全サブタスクが todo → 親タスクは todo のまま

### 2.3 ワークスプリットビュー

サブタスクを視覚的に工数配分で表示:

```
親タスク: LP制作 [見積: 40h] [担当: チーム]
├── デザイン        [佐藤] 16h ████████████████░░░░ 80%  4/5〆
├── コーディング     [田中] 12h ████████░░░░░░░░░░░░ 40%  4/7〆
├── コンテンツ作成   [鈴木]  8h ████████████████████ 100% ✓
└── テスト・修正     [渡辺]  4h ░░░░░░░░░░░░░░░░░░░░  0%  4/10〆
                          ────
                    合計: 40h (100% 配分済み)
                    進捗: 62% (加重平均)
                    残り: 15.2h
```

### 2.4 工数アラート

| 条件 | アラート |
|------|---------|
| サブタスク工数合計 > 親タスク見積 | 🔴 「見積超過: +Xh」 |
| サブタスクが未アサイン | 🟡 「未アサインのサブタスクがあります」 |
| サブタスク締切 > 親タスク締切 | 🟡 「親タスクの締切を超えるサブタスクがあります」 |
| 担当者の稼働率 > 80% | 🟡 「{name}さんの稼働率が高くなります」 |

### 2.5 ガントチャート連携

サブタスクが設定されている場合、ガントチャートで:
- 親タスクはサマリーバー（全サブタスクの期間をカバー）
- サブタスクは個別バー（担当者名付き）
- 依存関係は矢印で表示

## 4. データモデル

### 4.1 既存フィールドの活用

サブタスクは既存の `tasks` テーブルの行（`parent_task_id` が設定済み）。
追加のDBカラムは不要 — 既存フィールドを活用:
- `estimated_hours` → サブタスク個別見積
- `actual_hours` → サブタスク実績
- `confirmed_deadline` → サブタスク個別締切
- `priority` → サブタスク個別優先度
- `assigned_to` → サブタスク担当者
- `description` → 作業詳細

### 4.2 依存関係（新規テーブル）

サブタスク間の依存関係のみ新規テーブルが必要:

```sql
CREATE TABLE task_dependencies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  depends_on_task_id UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(task_id, depends_on_task_id),
  CHECK (task_id != depends_on_task_id)
);
```

### 4.3 親タスク自動更新ロジック

サブタスク更新時に親タスクを更新するトリガー:
- フロントエンド側で計算（DBトリガー不要）
- `useSubtasks` の結果から進捗を算出し、親タスクの `progress` を更新
- 更新タイミング: サブタスクの status / progress / estimated_hours 変更時

## 6. PMのワークフロー

```
1. タスク受領
   └→ 「LP制作 40h」を受け取る

2. ワークスプリット
   └→ サブタスクに分割:
      - デザイン (16h, 佐藤, P2, 4/5)
      - コーディング (12h, 田中, P2, 4/7)
      - コンテンツ (8h, 鈴木, P3, 4/4)
      - テスト (4h, 渡辺, P3, 4/10)

3. バランス確認
   └→ 合計40h = 見積40h ✅
   └→ 全員の稼働率チェック ✅

4. 進捗モニタリング
   └→ サブタスク完了で親タスク進捗が自動更新
   └→ 遅延サブタスクがアラート表示

5. 完了
   └→ 全サブタスク完了 → 親タスク自動完了提案
```

## 7. レビュー

### PM（プロジェクトマネージャー）
工数配分の可視化と自動進捗計算は日常業務の効率を大きく改善する。特にワークスプリットビューは、クライアントへの進捗報告にもそのまま使える。稼働率アラートがあることで、特定メンバーへの過負荷を事前に防げる点も重要。

### Tech Lead
既存の `tasks` テーブルのフィールドを活用する設計は、マイグレーションリスクが低く実装しやすい。依存関係テーブルのみ新規追加だが、シンプルな構造で十分。進捗計算をフロントエンドに寄せることで、DBトリガーの複雑さを回避している点は妥当。ただし、サブタスク数が多い場合のパフォーマンスは監視が必要。

### UX
サブタスク追加フォームは現行のシンプルさを維持しつつ必要項目を追加している。ワークスプリットサマリーカードは情報密度が高いが、プログレスバーと色分けで直感的に理解できる。モバイル表示時のレイアウト崩れには注意が必要。

### Frontend
Phase 1 は既存コンポーネントの拡張で対応可能。`useSubtasks` フックに計算ロジックを集約し、親タスク更新は `useMutation` のonSuccess内で実行する。サマリーカードは Recharts の積み上げバーチャートで実装可能。

### Backend
新規テーブルは `task_dependencies` のみ。RLS ポリシーは `tasks` テーブルと同一の権限モデルを適用すれば十分。親タスクの自動更新をフロントエンドで行う設計のため、API変更は最小限。将来的にリアルタイム同期が必要になった場合は Supabase Realtime のサブスクリプションを追加する。
